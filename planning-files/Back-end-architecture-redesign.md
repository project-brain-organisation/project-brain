Architectural Specification: Project Brain

System Overview & UI Blueprint Project Brain is a visual mind-mapping and knowledge management application consisting of a three-panel user interface powered by a graph-structured data engine and AI-assisted tooling: Left Sidebar Panel: Administrative navigation, multi-tenant project selectors, and profile configurations. Middle Panel (Feed): A text/markdown block editor displaying a linear feed of user thoughts and nested sub-thoughts. Right Panel (Canvas): A visual network graph rendering thoughts and labels as interactive network nodes, and their relationships as directed structural edges. Autonomous Extension Layer: A Model Context Protocol (MCP) server running as a sidecar process to allow AI agents to navigate, manipulate, and contextually interact with the user's graph environment using standard HTTP streaming.
Database Architecture (Drizzle ORM / PostgreSQL) The schema implements the Entity Supertype pattern (Table-per-Type inheritance) paired with an advanced Multi-Tenant Composite Foreign Key enforcement strategy. This configuration eliminates join overhead for standard workspace reads, guarantees absolute tenant isolation at the database layer, and models the project itself as the literal root node of its own graph network. Table Relationships & Architectural Enforcement Central Entity Table: Acts as the ultimate source of truth and global registry for every node in the graph (Projects, Thoughts, and Labels). It generates the base IDs and tracks the top-level project boundary for every record. It includes a self-referencing constraint so that the project root node points to itself. Subtype Data Tables (project_meta, thoughts, labels): These tables hold the unique, concrete data payloads for each node type (e.g., text content for thoughts, names/colors for labels, user ownership/privacy toggles for projects). They inherit their primary identifiers directly from the central entity table. Composite Foreign Key Trick: The relationship (edges) table and subtype tables duplicate the project identifier. By binding the node ID and project ID together into a composite foreign key constraint, the database itself mathematically guarantees that an edge can never link a node from Project A to a node from Project B. It also allows for seamless cascading deletesвЂ”deleting a project node instantly wipes out all associated thoughts, labels, and connections. Vector Sub-System (chunks): Houses the semantic fragments of broken-down text alongside float arrays for vector searches. It maps to thoughts on a many-to-one basis and uses the composite foreign key trick to guarantee that search text chunks never drift outside their secure project boundary.
NestJS Backend Modular Architecture To keep files clean, decoupled, and highly performant, the app uses a Layered Domain Architecture rather than table-by-table modules. Highly volatile structural graph interactions are grouped into abstractions to protect the system from circular reference compile errors. Modular Dependency Mapping DatabaseModule: A globally shared configuration manager initializing the connection pool and exporting the core Drizzle client instance to all feature boundaries. AuthModule: The security validation gateway mapping passport and token operations. It handles hashing and session tracking, depending strictly on the user module to verify profiles. UsersModule: An isolated user account registry providing basic profile data management (creation, lookups, updates) without taking on security or authentication processing bloat. ProjectsModule: Handles the left sidebar configuration details (project creation, names, and public/private visibility toggles). It imports the workspace module to instantiate project roots as nodes during initialization. WorkspaceModule: Encapsulates the entire domain context of the active working workspace (middle feed and right canvas). It bundles text thoughts, tagging arrays, background semantic vector parsing pipelines, and real-time update pathways under one unified feature umbrella to ensure clean data transitions.

Model Context Protocol (MCP) Integration Strategy The sidecar MCP server operates purely as an Orchestrator and Protocol Translator. It exposes JSON tool definitions to the LLM agent, but contains no database connections, direct business logic, or local validation schemas. Core Execution Flow for AI Tools: LLM Call Triggered: An LLM agent (e.g., Cursor, Claude) requests a tool execution like create_thought. Context Injection: The MCP Sidecar accepts the instruction and extracts the active user's HTTP authorization token from the active session runtime context. Controller Delegation: The MCP server uses a standard network request to pass the data payload directly to the existing NestJS Backend Controller endpoints. Enforced Safety Verification: The NestJS controller routes the incoming tool request through its standard security layer and schema parsers, guaranteeing that the AI agent can never read, modify, or leak cross-project data boundaries. Markdown Response Formatting: The NestJS controller processes the write or read query and returns a standard response, which the MCP server formats into a clean Markdown text block for the LLM to interpret. Core Architectural Gains: Zero Bloat in NestJS: Your backend remains a pure, high-performance web service without importing any LLM SDKs, prompt engineering libraries, or MCP tools. Streamable HTTP (SSE): By using the official protocol SDK, the sidecar app natively binds to standard stream transports to handle long-running LLM agent tool loops seamlessly. Unified Validation: Your input schemas validation rules remain unified. Changes made to a database constraint automatically cascade up to your API endpoints, ensuring both users and AI tools follow the exact same application rules.
